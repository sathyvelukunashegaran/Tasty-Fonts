# Product

## Register

product

## Users

Tasty Custom Fonts is for WordPress implementers who need dependable typography control inside real client sites: site builders, agencies, designers who work in WordPress, and technically confident site owners. They are usually in the WordPress admin, balancing creative font choices with delivery, compliance, editor sync, and builder compatibility. They want to add or import families, choose delivery profiles, preview draft role assignments, publish safely, and trust that Gutenberg, page builders, generated CSS, and runtime loading stay aligned.

## Product Purpose

Tasty Custom Fonts gives WordPress users one place to manage every font source and delivery path. It unifies local uploads, Google Fonts, Bunny Fonts, Adobe Fonts, role assignments, generated CSS, Gutenberg presets, builder integrations, diagnostics, and site-transfer tooling in a single admin workspace.

Success means a user can understand what fonts are available, what will load on the frontend, what is still draft, and what needs attention without juggling plugins, code snippets, CDN settings, or fragile manual CSS. The product should make sophisticated font operations feel controlled, reversible, and WordPress-native.

## Brand Personality

Precise, calm, professional.

The product voice should feel expert without being cold. It should reduce anxiety around typography changes, self-hosting, and compliance by using plain language, clear state, and confident workflows. It can feel polished and premium, but it should never feel decorative for its own sake.

## Anti-references

Avoid generic WordPress settings pages that feel like long option dumps, bloated font plugins that expose implementation clutter before user intent, noisy SaaS dashboards with decorative gradients and empty metrics, and AI-looking interfaces that overuse cards, badges, glow, or fake excitement.

Do not copy Stripe, Linear, Figma, Notion, Raycast, or WordPress admin directly. Use them only as quality references for density, alignment, keyboard accessibility, restrained feedback, and interaction discipline.

Avoid visual systems that rely on red/green status color alone, nested cards, modal-first workflows, or marketing-page drama inside the admin workspace.

## Design Principles

1. **Font work first.** Imported families, delivery profiles, role assignments, generated CSS, and diagnostics must be easier to scan than the surrounding chrome.
2. **Draft before live.** The interface should make publish state obvious and keep risky typography changes reviewable, reversible, and deliberate.
3. **WordPress-native confidence.** Controls, copy, permissions, focus states, and admin behavior should feel at home in WordPress while looking more refined than a default settings screen.
4. **One source of truth.** Library data, runtime CSS, Gutenberg presets, builder integrations, and support diagnostics should read as connected outputs from the same system.
5. **Compliance without fear.** Self-hosting, CDN delivery, preload, preconnect, and provider choices should be understandable in practical site terms, not hidden behind jargon.

## Accessibility & Inclusion

Target WCAG AA for product UI. Preserve keyboard-first navigation, visible focus states, semantic WordPress controls, reduced-motion behavior, readable dense text, and status communication that does not depend on color alone. Design for users managing client sites under time pressure, including users who need compact scanning, clear labels, predictable confirmation steps, and accessible diagnostics they can copy or share.
